# Чеклист за тестване — Банкова система

Тестови акаунти: `admin/admin123`, `employee/employee123`, `client1/client123`.
Адрес: `http://localhost:8083`.

Легенда: ⬜ = за тестване.

---

## 1. Автентикация (вход / изход / регистрация)

- ⬜ Вход с `admin/admin123` → успех, пренасочване към началната страница
- ⬜ Вход с `employee/employee123` → успех
- ⬜ Вход с `client1/client123` → успех
- ⬜ Грешна парола → връща `/login?error` със съобщение „Грешно потребителско име или парола"
- ⬜ Празни полета → браузърна валидация (required)
- ⬜ Изход (бутон „Изход") → `/login?logout` със съобщение
- ⬜ Достъп до защитена страница без вход (напр. отвори `/clients` в нов прозорец) → пренасочване към `/login`

### Регистрация на нов клиент
- ⬜ `/register` се отваря без вход
- ⬜ Успешна регистрация (валидни данни) → `/login?registered`, после вход с новите данни работи
- ⬜ Различни пароли в двете полета → грешка „Паролите не съвпадат"
- ⬜ Парола под 4 символа → валидационна грешка
- ⬜ Празни задължителни полета (име, фамилия, ЕГН, потребител) → грешки под полетата
- ⬜ ЕГН различно от 10 цифри → грешка
- ⬜ Дублирано потребителско име → грешка „вече съществува"
- ⬜ След регистрация новият потребител влиза и вижда **Моят профил** с празни сметки/кредити

---

## 2. Роли и защита (авторизация)

### admin
- ⬜ В навигацията вижда: Клиенти, Сметки, Кредити, **Кредитни видове**
- ⬜ Вижда бутоните за изтриване (червени) навсякъде

### employee
- ⬜ В навигацията вижда: Клиенти, Сметки, Кредити — **НЕ** вижда „Кредитни видове"
- ⬜ Ръчно отваряне на `/credit-types` → достъпът е отказан (пренасочване към началото)
- ⬜ Бутоните за изтриване **не** се виждат
- ⬜ Опит за изтриване по URL (POST `/clients/{id}/delete`) → блокирано на ниво услуга (страница „Нямате права")

### client
- ⬜ В навигацията вижда само **Моят профил**
- ⬜ Ръчно отваряне на `/clients`, `/accounts`, `/credits`, `/credit-types` → пренасочване към началото (отказан достъп)
- ⬜ Вижда само собствените си сметки и кредити, не чужди

### API защита (по желание, с curl/Postman, влязъл като съответната роля)
- ⬜ `GET /api/clients` като client → връща само собствения клиент, не всички
- ⬜ `GET /api/credits` като client → само собствените кредити
- ⬜ `POST /api/credits` като client → `403 Forbidden` (JSON `{"error":"Нямате права за достъп"}`)
- ⬜ `DELETE /api/clients/{id}` като employee → 403

## 3. Клиенти (admin / employee)

- ⬜ Списък с клиенти се зарежда; колоните „Сметки" и „Кредити" показват правилни бройки
- ⬜ **Добавяне на физическо лице** с валидни данни → появява се в списъка с бадж „Физическо"
- ⬜ Физическо лице: празни полета → валидационни грешки под полетата
- ⬜ Физическо лице: ЕГН ≠ 10 цифри → грешка
- ⬜ **Добавяне на юридическо лице** с валидни данни → бадж „Юридическо"
- ⬜ Юридическо лице: празни полета (фирма, ЕИК, представител) → грешки
- ⬜ **Редактиране** на физическо лице → промените се запазват
- ⬜ **Редактиране** на юридическо лице → промените се запазват
- ⬜ **Изтриване** на клиент (само admin) → изчезва от списъка
- ⬜ Изтриване на клиент със сметки/кредити → каскадно се трият и те (потвърди в Сметки/Кредити)
- ⬜ Свързване чрез „Потребителско име": задай `client1` на клиент → влез като `client1` → виждаш данните му в Моят профил

---

## 4. Банкови сметки (admin / employee)

- ⬜ Списък със сметки се зарежда (вкл. seed сметките)
- ⬜ **Откриване на сметка**: избор на притежател + начална наличност → IBAN се генерира автоматично, статус „Активна"
- ⬜ Откриване без избран притежател → валидационна грешка
- ⬜ Откриване с отрицателна наличност → грешка
- ⬜ Генерираните IBAN-и са уникални (открий няколко)
- ⬜ **Закриване** на сметка с нулева наличност → статус „Закрита", бутонът „Закрий" изчезва
- ⬜ Закриване на сметка с ненулева наличност → бизнес грешка („Сметка с ненулева наличност не може да бъде закрита")
- ⬜ **Изтриване** на сметка (само admin)

---

## 5. Кредитни видове (само admin)

- ⬜ Списък показва seed-натите „Потребителски" и „Ипотечен" с лихва/лимити
- ⬜ **Създаване** на нов вид с валидни данни → появява се
- ⬜ Дублирано име → грешка „вече съществува" (409)
- ⬜ Лихва ≤ 0 → валидационна грешка
- ⬜ Максимална сума ≤ 0 → грешка
- ⬜ Максимален срок < 1 → грешка
- ⬜ **Редактиране** на параметри → запазват се
- ⬜ Преименуване към съществуващо име → грешка
- ⬜ **Изтриване** на вид

---

## 6. Кредити + погасителен план (ядрото)

### Отпускане
- ⬜ **Отпускане на кредит**: клиент + вид + сума + срок → кредит със статус „Активен" и генериран план
- ⬜ Сума над максималната за вида → бизнес грешка с лимита
- ⬜ Срок над максималния за вида → бизнес грешка с лимита
- ⬜ Сума ≤ 0 → валидационна грешка
- ⬜ Срок < 1 → валидационна грешка
- ⬜ Без избран клиент/вид → валидационни грешки

### Проверка на математиката (отвори „План" на кредит)
Пример за проверка: 10000 лв., „Потребителски" (8.5%), 12 месеца:
- ⬜ Месечната вноска е постоянна за всички месеци (~872.20 лв.; последната може да е с няколко стотинки разлика)
- ⬜ Първият месец има **най-голяма лихва** и най-малка главница
- ⬜ Към края лихвата намалява, главницата нараства
- ⬜ „Остатък" на последния ред е **0.00**
- ⬜ Сборът на „Главница" по всички редове = отпуснатата сума (10000.00)
- ⬜ За всеки ред: Вноска = Главница + Лихва
- ⬜ Полето „Общо за връщане" > отпуснатата сума (с лихвата)
- ⬜ Ипотечен (3.2%) срещу Потребителски (8.5%) за същата сума → различна вноска и обща лихва

### Плащане на вноски и статус
- ⬜ **Плащане** на първата вноска → редът става зелен, „Платени" нараства (1/12)
- ⬜ Опит за плащане на по-късна вноска преди по-ранна → грешка „Първо платете предходните"
- ⬜ Опит за повторно плащане на платена вноска → грешка „вече е платена"
- ⬜ „Остатък за плащане" в обобщението намалява след всяко плащане
- ⬜ Плащане на **последната** вноска → статус на кредита става „Изплатен" (зелен бадж)
- ⬜ **Изтриване** на кредит (само admin)

## 7. Клиентско самообслужване (`/my`, роля client)

- ⬜ Вход като `client1` → таблото показва име, ЕГН и собствените сметки/кредити
- ⬜ Сметките и кредитите са само на този клиент (не чужди)
- ⬜ Бутон „План" → отваря погасителния план на собствен кредит
- ⬜ В клиентския изглед **няма** бутони „Плати" (само за четене)
- ⬜ Опит за чужд кредит по URL (`/my/credits/{чужд_id}`) → „Кредит … не е намерен"

---

## 8. Валидиране на данни (общо)

- ⬜ Всички съобщения за грешки са на български и се показват под съответното поле
- ⬜ При грешка формата се презарежда и **запазва** въведените стойности
- ⬜ Числовите полета (сума, лихва, срок) не приемат текст

---

## 9. Управление на изключения

### Изгледи (HTML страница за грешка)
- ⬜ Отваряне на несъществуващ кредит `/credits/999999` → страница „Записът не е намерен" (404)
- ⬜ Нарушено бизнес правило (напр. закриване на сметка с пари) → страница „Нарушено бизнес правило" (422)
- ⬜ Отказан достъп → пренасочване/страница „Нямате права за достъп" (403)

### REST API (JSON)
- ⬜ `GET /api/credits/999999` → 404 с JSON (`timestamp, status, error, message, path`)
- ⬜ `POST /api/credit-types` с невалидно тяло → 400 с поле `fieldErrors`
- ⬜ Дублиран кредитен вид през API → 409
- ⬜ Сума над лимита през API → 422

## 11. Управление на служители (само admin)

- ⬜ Таб „Служители" и карта на началния екран се виждат **само** от admin
- ⬜ Списъкът показва seed-натия `employee` с роля и статус „Активен"
- ⬜ **Създаване** на нов служител (валидно потребителско име + парола) → появява се със статус „Активен"; toast „Записът е добавен успешно"
- ⬜ Нов служител може да влезе с зададените данни и вижда Клиенти/Сметки/Кредити (но не „Кредитни видове"/„Служители")
- ⬜ Различни пароли в двете полета → грешка „Паролите не съвпадат"
- ⬜ Парола под 4 символа → валидационна грешка
- ⬜ Дублирано потребителско име → грешка „вече съществува"
- ⬜ **Деактивиране** на служител (с потвърждение) → статус „Неактивен"
- ⬜ Деактивиран служител **не може да влезе** (грешка при логин)
- ⬜ **Активиране** обратно → пак може да влиза
- ⬜ **Изтриване** на служител (с потвърждение) → изчезва от списъка
- ⬜ Опит за достъп до `/employees` като employee или client → отказан (пренасочване)
